# ADR-0000: Initial Architectural Decisions

**Status:** Accepted
**Date:** 2026-05-25
**Context:** Q&A rounds 1–4 between Heiko (architect) and Claude before
any code was written. These decisions form the foundation referenced by
the Smart Sentinel Eye constitution.

Each row below is a discrete decision. Subsequent ADRs supersede or
extend these as the system evolves. Use the ID column to reference them
from specs, plans, and code comments.

| # | Decision | Status |
|---|---|---|
| 001 | Frontend = React (browser-only); Backend = .NET / C#; DDD with value objects throughout; CQRS / event sourcing only where justified. | Locked |
| 002 | Scale: 20-camera pilot, 250-camera production target. Recording is out of scope for v1 but must not be precluded by architecture. | Locked |
| 003 | Workflow: informal Q&A rounds → Spec-Kit (`specify` / `plan` / `tasks`) → GitHub Project board. | Locked |
| 004 | (Superseded by 015) Streaming = WebRTC end-to-end with sub-500 ms glass-to-glass target. StreamKeeper acts as SFU. Overlays composited client-side, timed to RTP timestamps. | Superseded |
| 005 | Camera protocols: RTSP + ONVIF (Profile S/T) on day one. Adapter pattern in StreamKeeper for vendor-specific drivers (Axis VAPIX, Hikvision, Bosch, Hanwha…) later, without touching the core. | Locked |
| 006 | Deployment: on-prem first per fab (v1 must be fully self-contained, no cloud dependency). Hybrid cloud control plane is the v2 competitive differentiator. v1 services must use idempotent IDs and versioned config so cloud sync is additive — not a rewrite. | Locked |
| 007 | Identity = Keycloak per fab (self-hosted OIDC). v2 cloud control plane will federate to each fab's Keycloak. | Locked |
| 008 | Kiosk auth = device-bound credential → OIDC `client_credentials` → short-lived token, view-only scope. Operators authenticate separately (auth code flow) and bind to a kiosk to gain control scopes (PTZ, camera change, layout switch). No PTZ without an operator token. | Locked |
| 009 | Persistence default = PostgreSQL. **Marten** (Postgres-backed event sourcing) only where a context's invariants justify it — first candidates Overlays, Automation. **Prometheus** for metrics. **MinIO** as the future object store. No EventStoreDB. | Locked |
| 010 | Messaging = RabbitMQ for internal service-to-service messaging and external system ingress. REST API also exposed for external publishers — translates to RabbitMQ publishes. Plan for Streams plugin if replay is required. | Locked |
| 011 | Transcoding = passthrough when camera profile is WebRTC-compatible; GPU transcode (NVENC / Intel Quick Sync) only when forced (H.265, exotic profile). Sizing budget: ~1 NVENC-class GPU per 50–100 transcodes. | Locked |
| 012 | StreamKeeper scaling = horizontal shard-by-camera. Coordinator service (Raft/etcd-class consistency) owns cam→SFU ownership map. Clients query the coordinator. Failover under 5 s. | Locked |
| 013 | Network = cameras on isolated OT VLAN. StreamKeeper is dual-NIC and the *only* bridge to the IT VLAN. No other service touches a camera directly. | Locked |
| 014 | Video-wall sync = frame-accurate via PTP (IEEE 1588) + coordinated playout. StreamKeeper emits presentation timestamps; kiosks buffer ~100–200 ms and present at T. Inter-display skew target < 5 ms. Deployment prerequisite: PTP-aware switches in the fab OT network. | Locked |
| 015 | **Supersedes ADR-004.** End-to-end latency SLO: external event arrival → overlay rendered, frame-synced = **≤ 800 ms** (was ≤ 500 ms). WebRTC still the streaming protocol. Sub-SLOs: cam→SFU ≤80 ms, SFU→kiosk decode ≤120 ms, presentation buffer ≤200 ms, event→overlay state ≤200 ms, composite+render ≤50 ms, headroom ≤150 ms. | Locked |
| 016 | Bounded contexts: 9 — Camera Catalog, Stream Distribution, Layout & Composition, System Variables, Event Ingestion, Overlay Designer, Automation, Identity & Authorization, Audit & Observability. Camera Catalog ≠ Stream Distribution (different lifecycles). Variables + Events + Overlays are three separate contexts coupled by RabbitMQ domain events; browser-side overlay engine reads from a per-kiosk WebSocket gateway. | Locked |
| 017 | System Variable types (v1) = `text`, `boolean`, `integer`, `decimal`, `datetime`, `json` (opaque). Each variable declares exactly one type. JSON variables are stored and forwarded as opaque strings; overlay expressions parse them client-side. Promoting `json` to `json + schema` is reserved for a later version. | Locked |
| 018 | External Event ingestion = hybrid registration model. Each source has a `strict` flag (rejects unknown event types, schema-validates known) or `discovery` flag (accepts unknown, quarantines them in an inspector UI for promotion to the registry). Validated events feed rules / overlays; quarantined events are audit-only. | Locked |
| 019 | Overlay expression language = **CEL (Common Expression Language)** via a .NET implementation. Pure expressions, typed, sandboxed, deterministic, no I/O. Used for text content, visibility predicates, color bindings, JSON deep-access. The same language is reused inside automation rule `where` clauses. | Locked |
| 020 | Automation engine v1 = declarative rules (YAML/JSON documents with fixed trigger + action schema, CEL conditions, cooldowns, priority/conflict policy) + builder UI. **Forward-compat constraint:** Automation exposes `IRuleEngine` as a strategy interface; rule definitions are engine-tagged so a v2 visual-workflow engine (Node-RED / n8n-style) can plug in. Rule execution always issues commands via RabbitMQ — never direct DB writes. | Locked |
| 021 | Event time reference = hybrid. Use `source.ts` if present AND `|source.ts − now| ≤ 5 s`, else fall back to ingestion time and mark `time_uncertain`. Each event persists `time_basis ∈ {source, ingestion}` and `used_ts`. Overlay engine renders at the frame whose presentation timestamp matches `used_ts`. Inspector UI surfaces the basis. | Locked |
| 022 | Draft → preview → publish workflow applies to **Overlays**, **Layouts** (with assignment guard + render-error fallback), **Automation rules** (with dry-run mode), and **Camera Catalog** (staged config changes applied together). All other contexts edit live with audit log. Common publish pipeline: draft → validate → preview/dry-run → publish → atomic activate → audit trail. | Locked |
| 023 | Authorization = **fixed RBAC for v1** with 4 roles (admin, operator, viewer, kiosk) and per-context scope bundles. Roles mapped to Keycloak realm roles; customer-specific role variants handled via Keycloak groups. All authz checks go through `IAuthorizationDecisionPoint` so a policy engine (OPA / Cedar / ABAC) can plug in for v2 without refactoring call sites. | Locked |
| 024 | **.NET Aspire** is the orchestration and composition layer. AppHost declares all resources in C#. Aspire dashboard provides dev-time telemetry. Production deployment via `aspire publish` generating Helm charts for k3s. Aspire integrations preferred over ad-hoc configuration. | Locked |
| 025 | Production orchestration target = **k3s + Helm** generated by Aspire's Kubernetes publisher. Per-fab 3-node k3s control plane plus GPU-labelled worker nodes for StreamKeeper SFU DaemonSets. Pilot fab also runs k3s (single-node if necessary) to keep one toolchain. v2 cloud control plane will use Argo CD / Flux to push Helm releases to per-fab git repos. | Locked |
| 026 | Observability = OpenTelemetry Collector fanning OTLP to **both** Aspire dashboard AND Grafana stack (Prometheus + Loki + Tempo + Grafana + Alertmanager) during walking-skeleton + first two features. Comparison phase ends before v1 GA — single sink committed at that point (expected: Grafana wins prod, Aspire stays for dev). ADR carries an explicit sunset clause. | Locked |
| 027 | Repo + solution layout = **single monorepo, single .NET solution**. `src/` (one project per bounded context with Domain / Application / Infrastructure / Api split, plus `AppHost`, `ServiceDefaults`, `Shared.Kernel`, `Shared.Contracts`), `apps/web/` (React, Aspire JS resource), `tests/`, `deploy/helm/`, `specs/`, `docs/adr/`. Cross-context boundaries enforced by NetArchTest. Communication between contexts only via `Shared.Contracts` — no direct project references. | Locked |
